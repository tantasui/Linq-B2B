01 / Executive Overview

What Linq Does

Linq lets a Nigerian merchant accept USDT or USDC while thinking in NGN. The merchant creates open, fixed, or static receive surfaces. Customers provide name and email, select token and network, and receive exact payment instructions.

The key separation is custody. Dynamic manages merchant wallet control. Linq stores public wallet routing metadata only. Paycrest creates settlement orders and provides receive addresses for payment/request links.

  • NGN-first payment links and dashboard totals.
  • Paycrest rate refresh at payment time.
  • Payer details stored against each order.
  • 15-minute cache for Paycrest tokens and rates.
  • Static wallet mode separated from Paycrest order mode.
  • Retry transfer uses the verified merchant bank account only.

Implementation Status

Implemented

Plain SQL schema, server modules, API routes, checkout order flow, merchant onboarding UI, order dashboard, webhook handler, cache abstraction, and report artifacts.

Env-Gated

Paycrest, Redis, Koyeb Postgres, and Dynamic are activated by environment variables. Local mode keeps the app demonstrable without live credentials.

Caveat

The Dynamic SDK install hung in this environment, so the app uses a bridge provider abstraction with deterministic fallback wallets. The bridge is the intended SDK swap point.

02 / System Architecture

Layered Design

The backend can now deploy separately to Koyeb from the backend/ directory using backend/Dockerfile. The frontend can call the public Koyeb backend URL while the backend connects to Koyeb Postgres through DATABASE_URL.

Presentation Layer

Next.js pages and React components render the mobile-first dashboard, receive page, settings onboarding, and checkout. Business UI stays compact and operational.

Backend API Layer

Route handlers under src/app/api expose onboarding, merchant, bank, Paycrest utility, payment link, order, retry, and webhook endpoints.

Server Domain Layer

src/server centralizes env parsing, validation, Paycrest requests, Redis cache behavior, webhook security, API response envelopes, and store access.

Data Layer

Koyeb Postgres accessed through direct SQL and the pg client is the intended production store. The current local store mirrors repository behavior so flows can work without a database URL.

05 / Checkout Flow

Customer Payment Sequence

1

Checkout opens the payment link and fetches merchant/link data from GET /api/payment-links/[id].

2

The payer's name and email are read from localStorage or collected once.

3

Checkout loads Paycrest-supported tokens and networks from the backend.

4

The payer selects USDT or USDC and a supported network.

5

The backend refreshes the NGN rate for the chosen token/network/amount.

6

The app creates a local order, then creates a Paycrest order.

7

Checkout displays exact crypto amount, receive address, QR code, Paycrest id, status, and expiry if provided.

Why Local Order First

Creating the local order before exposing transfer instructions gives Linq an internal audit trail even if the provider request, webhook, or customer session fails after the first attempt.

06 / Paycrest Integration

Provider Responsibilities

  • Fetch supported tokens through /tokens.
  • Fetch rates through /rates/{network}/{token}/{amount}/{fiat}.
  • Fallback to provider rate bands through /provider/rates/{token}/{fiat}.
  • Verify bank accounts through /verify-account.
  • Create crypto-to-fiat orders through /sender/orders.
  • Normalize statuses from Paycrest into internal order states.

07A / Refunds and Retry Transfers

Refund and Recovery Flow

Refund Sequence Diagram

Paycrest Status

Paycrest sends refunding, refunded, failed, cancelled, or expired status.

Webhook Handler

Linq verifies signature, stores raw event, and normalizes the status.

Order Update

The order record changes state and becomes visible in dashboard polling.

Merchant Review

Merchant expands the compact order row to inspect payer, token, rate, and provider details.

Retry Eligibility

Retry is only shown for failed, refunded, expired, or cancelled statuses.

Verified Bank Only

Retry creates a TransferAttempt using the already verified payout account.

Why Retry Is Restricted

A payout retry can move money, so it must not let a merchant swap account numbers during a failure. The retry action always reuses the stored verified bank account to reduce fraud and support operational review.

Audit Trail

Every webhook creates an OrderEvent. Every retry creates a TransferAttempt with attempt number, status, reference, payload, and error message when applicable.

09 / Security

OWASP-Aligned Controls

Validation

All write routes parse request data through Zod. Schemas trim text, normalize email, restrict token and network values, bound amount size, and limit field lengths.

Webhook Integrity

Paycrest webhooks use HMAC SHA-256 verification with timing-safe comparison when a webhook secret is configured.

Rate Limiting

Bank verification and public order creation have rate limits. Production should move this to Redis or edge middleware.

Secrets

Paycrest API keys, webhook secrets, database URLs, Redis tokens, and Dynamic server tokens stay in environment variables and never in client bundles.

Custody

Linq stores public wallet addresses only. It never requests or stores private keys, seed phrases, MPC recovery shares, or Dynamic recovery secrets.

Authorization

Production should bind merchant routes to server-verified Dynamic sessions so one merchant cannot access another merchant's links, orders, or bank records.
